# Database Design Plan

## Database
- Engine: SQL Server
- Purpose: persistent storage for accounts, profiles, media metadata, discovery, messaging, credits, payments, safety, and admin actions.
- All registered platform users are called Customers.
- All company workers are called Employees.
- SQL Server stores all relational data.
- Some tables may intentionally include redundant/denormalized data for efficiency, as long as the source-of-truth and update rules are documented.

## Naming Rules
- Customer-facing user tables use `Customer` prefix.
- Company worker tables use `Employee` prefix.
- Primary keys should be system-generated UUID values unless a later design decision says otherwise.
- Time columns use UTC.
- Boolean state columns such as `Active` should have clear business meaning.
- Free-text notes use `Remark` when they are internal/admin-facing.

- Active
- Seed

- CreditsRemain
- TotalCharged

Notes:
- The requested customer `Password` is stored only as `PasswordHash`; the original password is never stored.
- Registration requires country, state/province, and city.
- A new real customer receives a sex-based illustrated default profile photo
  when no photo is uploaded.
- `ProfileCompleted = 0` directs the customer to Me after sign-in until all
  required profile sections have been saved.
- The prototype stores bounded string selections and photo references as JSON
  arrays on `CustomerProfile`; production may normalize them into child tables
  when reporting or localization requires it.
- `Seed` identifies the customer operating type and is an integer, not a bit.
- All seed customer profiles are generated by AI before the product is deployed.
- Seed profiles must be original synthetic characters or use explicitly licensed/commissioned assets; copied or lightly altered real-person profiles are prohibited.
- Seed biographies target 80-180 words and must remain under 500 words and 4,000 characters.
- Seed generation rights, model versions, originality checks, adult-appearance
  review, independent approval, and profile-presentation policy version are
  stored in `SeedProfileProvenance`.
- `Seed = 0` means a normal registered customer.
- `Seed = 1` means an employee-operated seed customer.
- `Seed = 2` means an autonomous robot customer.
- Allowed conversation pairs are real-real, real-seed, and real-robot.
  Seed-seed, seed-robot, and robot-robot pairs are rejected.
- Robot customers do not receive EmployeeSeed or SeedWorkAssignment rows.
- `CreditsRemain` is the authoritative current spendable balance used for
  customer display and transaction eligibility. It is intentionally stored on
  this frequently read row.
- `TotalCharged` is an intentionally redundant summary field for fast display
  and filtering.
- `PrimaryPaymentMethodId` points to the customer's preferred saved payment method.
- `CreditLedger` is the immutable transaction history used to explain,
  audit, and reconcile every change to `CreditsRemain`.
- Every grant or deduction updates `CreditsRemain` and appends the related
  `CreditLedger` row in one SQL transaction.
- Logout, browser close, session expiry, and Frontend synchronization never
  write a customer-provided balance into this column.
- Read frequency may be reduced by retaining the latest committed balance in
  Frontend session state or a server-side cache. Every paid command still uses
  the SQL Server balance and atomic sufficient-balance update.
- The source of truth for customer charges should be `ChargeRecord`; this table stores a cached total charged amount.
- `Active` means the customer account is allowed to use the platform. Suspended/banned/deleted states may need a more detailed `Status` column later.

Customer card information requirement:
- Card details must not be stored directly in `CustomerProfile`.
- Customer saved-payment information belongs in the restricted `CustomerPaymentMethods` table.
- `CardNumber` is represented by a payment-provider token plus masked display values such as last four digits.
- `BankName`, `CardType`, and expiration information may be stored as non-secret payment metadata.
- `SecCode`/CVV/CVC must never be stored.
- Card password/PIN must never be stored.
- The customer's platform login password is represented only by `PasswordHash`; a plain `Password` column is forbidden.

Employee password rules:
- The requested employee `Password` is stored only as `PasswordHash`; the original password is never stored.

- Do not store `SecCode`/CVV in SQL Server.
- Do not store full card numbers unless a later PCI/payment design explicitly requires it and proper protections are approved.
- Store employee payout or payment details in a separate restricted table such as `EmployeePaymentAccounts`.
- Prefer payment-provider tokens, bank account tokens, masked card last 4 digits, and encrypted payout metadata.

### ChatRecords
Purpose: records conversations between customers, including text, pictures, gift icons, and credits consumed by a message.

Primary key:
- ChatRecordId: system-generated UUID

Proposed columns:
- ChatRecordId
- ChatTime
- SenderId
- ReceiverId
- Text
- Image
- CreditUsed
- CreateTime

Foreign keys:
- `SenderId` references `CustomerProfile.CustomerId`.
- `ReceiverId` references `CustomerProfile.CustomerId`.

Column notes:
- `ChatTime` is the customer-visible message timestamp in UTC.
- `Text` contains the message text and may be null when the record contains only an image or gift.
- `Image` stores a media reference, picture reference, or gift icon reference. The final design should store a URL/media ID rather than image binary data directly in this table.
- `CreditUsed` records the credits charged for this specific message or gift.
- A text message costs 5 credits and is limited to 60 words.
- An accepted image or sound message costs 10 credits.
- Accepted pictures reference a processed media object no larger than 100 x 100
  pixels, with aspect ratio preserved.
- Processed picture media objects use JPG or PNG content type. Safely readable
  source formats are converted before acceptance.
- Accepted voice messages reference a processed audio object; exact duration and
  byte limits remain pending.
- `CreditUsed` should be zero only for an explicitly free message or system event.
- Sender and receiver cannot be the same customer.
- At least one of `Text` or `Image` must be present.
- Chat records should be retained according to the final message retention and deletion policy.

### EmployeeSeed
Purpose: records which employee is responsible for each seed customer and the geographic area represented by that assignment. An employee may be a human worker or a robot employee.

Recommended primary key:
- EmployeeSeedId: system-generated UUID

Proposed columns:
- EmployeeSeedId
- EmployeeId
- CustomerId
- Active
- ActiveEndTime
- Country
- State
- City
- CreateTime
- UpdateTime

Foreign keys:
- `EmployeeId` references `Employees.EmployeeId`.
- `CustomerId` references `CustomerProfile.CustomerId`.

Business rules:
- `CustomerId` must reference a customer whose `CustomerProfile.Seed = 1`.
- Only large cities have seed customers.
- Each employee is assigned up to 1,000 seed customers.
- Each large city has 100 seed customers.
- Each city allocation contains 80 female seed customers and 20 male seed customers.
- All seed customer profiles are generated by AI before product deployment.
- Robot employees are represented in `Employees` and can receive EmployeeSeed assignments.
- `Active = 1` means the employee currently oversees that seed customer.
- `ActiveEndTime` is null while the assignment is active and is set when responsibility ends.
- A seed customer should have only one active employee assignment at a time.

Redundant location data:
- `Country`, `State`, and `City` intentionally duplicate location data from `CustomerProfile` for faster employee workload and geographic reporting.
- The assignment location should match the seed customer's profile location when the assignment is created.
- The design must define whether later profile location changes update existing EmployeeSeed rows or preserve assignment history.

Capacity model:
- Up to 1,000 assigned seed customers per employee.
- 20 seed profiles active online at one time for an employee.
- Up to 10 simultaneous real-customer conversations per employee.
- Maximum online time per seed profile is two hours per day.
- Earlier requirements stated 20 cities per employee and 100 seed customers per city, which equals 2,000 seed customers. That geographic rule conflicts with the newer 1,000-seed limit and must be reconciled.

### CreditLedger
Purpose: immutable transaction history for every customer credit increase and decrease.

Proposed columns:
- CreditLedgerId
- CustomerId
- TransactionTime
- TransactionType
- CreditsChange
- BalanceAfter
- ChargeRecordId
- ChatRecordId
- GiftTransactionId
- RelatedCustomerId
- EmployeeId
- ReasonCode
- CreateTime
- Remark

Transaction types include:
- CreditPurchase
- RegistrationReward
- TextMessageCharge
- MediaMessageCharge
- GiftSent
- GiftReceived
- RefundAdjustment
- ChargebackAdjustment
- AdminCorrection

Rules:
- RegistrationReward is granted once and only to eligible real customers.
- Credit deductions cannot create a negative balance unless a later policy permits debt.
- Ledger entries are append-only.
- `CustomerProfile.CreditsRemain` is the authoritative operational balance.
- `BalanceAfter` must equal the `CustomerProfile.CreditsRemain` value committed
  in the same transaction.
- Financial/message/gift operations, the `CreditsRemain` update, and their
  ledger entries must be atomic.
- Concurrent deductions use a database-enforced sufficient-balance condition
  so two accepted commands cannot overspend the same balance.
- A client balance is advisory at most. A mismatch returns the committed
  server balance and may create a diagnostic/reconciliation event; it cannot
  directly alter `CreditsRemain`.

- Sender deduction, recipient grant, platform share, and ledger records must be atomic.
- Successfully sent gifts are final and non-refundable.
- For a real recipient, 80% is added to the recipient Customer CreditLedger.
- For a seed recipient, 80% is added to EmployeeCreditLedger for the overseeing
  employee snapshotted at gift time.
- Later EmployeeSeed reassignment does not move completed gift credits.
- A seed gift is rejected before charging if no overseeing EmployeeId can be resolved.

## Data Integrity Rules
- Customer must have one CustomerProfile row.
- Employee must have one Employees row.
- Profile option joins must reference valid static options.
- Credit ledger is append-only.
- Payment events are append-only.
- Audit logs are append-only.
- Message deletion should be soft-delete by participant, not hard-delete globally, unless legally required.
- CustomerProfile.CreditsRemain is the authoritative current spendable balance
  and must reconcile with the latest CreditLedger.BalanceAfter and ledger sum.
- CustomerProfile.TotalCharged must reconcile with successful ChargeRecord rows.
- CustomerProfile.PrimaryPaymentMethodId, when present, must reference an active payment method belonging to the same customer.
- Employees.TotalEarn must reconcile with employee earnings/payout records once those tables are designed.
- CompanyDayMake must reconcile with its source ChargeRecord and credit-consumption records.
- EmployeeMonthReport must reconcile with attributed ChargeRecord and credit-consumption records.
- CompanyMonthReport.TotalMoney must reconcile with CompanyDayMake.
- CompanyMonthReport.TotalPaid must reconcile with paid EmployeeMonthReport rows.
- AdvertisementRecord expenses must reconcile with provider invoices and company expense reports.
- CloudService expenses must reconcile with provider invoices and company expense reports.
- Registration rewards, message charges, media charges, and gifts must reconcile with CreditLedger.
- GiftTransactions must reconcile sender cost, recipient 80% share, and platform 20% share.
- CompanyYearReport must reconcile with finalized monthly reports and approved annual expense totals.
- ChatRecords.SenderId and ChatRecords.ReceiverId must reference existing customers.
- EmployeeSeed.CustomerId must reference a seed customer.
- A seed customer can have no more than one active EmployeeSeed assignment.
- A seed profile can have no more than one active worker assignment.
- A real-customer conversation can have no more than one active worker assignment.
- Seed daily online time cannot exceed two hours.
